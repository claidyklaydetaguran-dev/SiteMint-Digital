1. SCHEMA — add to intakeFirms: stripeCustomerId (text, nullable),
   stripeSubscriptionId (text, nullable).

2. NEW ENV VAR — STRIPE_RECEPTIONIST_PRICE_ID. This will be set by the
   account owner after creating a recurring Price in their Stripe
   dashboard; do not hardcode a price or product ID anywhere. If it's
   missing at runtime, the upgrade endpoint should return a clear 500
   error message ("Billing is not configured yet") rather than crashing
   unhandled.

3. NEW ROUTE — POST /api/receptionist/billing/create-checkout-session
   (gated by requireReceptionistAuth). Creates a Stripe Checkout Session
   in subscription mode for the logged-in firm:
   - mode: 'subscription', line_items: [{ price:
     process.env.STRIPE_RECEPTIONIST_PRICE_ID, quantity: 1 }]
   - If the firm doesn't yet have a stripeCustomerId, create a Stripe
     customer first (email = firm's email) and save the id back to the
     intakeFirms row before creating the session; reuse it on subsequent
     checkouts.
   - success_url: back to /app/settings with a query param indicating
     success (e.g. ?upgraded=1); cancel_url: back to /app/settings.
   - Returns the Checkout Session URL for the frontend to redirect to.

4. WEBHOOK — find the existing Stripe webhook route if one exists and add
   a handler there; otherwise create a new one at, e.g.,
   /api/receptionist/billing/webhook. Must verify the Stripe signature
   using the raw request body and STRIPE_WEBHOOK_SECRET (same rigor as the
   existing Twilio signature validation elsewhere in this codebase — do
   not skip verification even in development). Handle:
   - checkout.session.completed (or customer.subscription.created) →
     look up the firm by stripeCustomerId, set planTier = 'paid', store
     stripeSubscriptionId.
   - customer.subscription.deleted (cancellation) → set that firm's
     planTier back to 'trial'. This is a real requirement, not scope
     creep — a canceled subscription must actually lose paid access.
   - Make the handler idempotent — Stripe can redeliver the same event;
     do not double-process (e.g. check current planTier before writing,
     or track processed event ids).

5. SETTINGS PAGE — replace the "coming soon" placeholder in
   ReceptionistSettings.tsx with a real Billing section:
   - If planTier is 'trial': show current usage (reuse the same
     count/limit data already used in the trial banner), and an "Upgrade
     to Pro" button that calls the new create-checkout-session endpoint
     and redirects the browser to the returned Stripe URL.
   - If planTier is 'paid': show "You're on the Pro plan" with no cap
     messaging, and a "Manage billing" link if a Stripe customer portal
     is easy to wire from the existing Stripe setup (billingPortal
     sessions) — if that's non-trivial given what's already in place,
     skip it for this pass and note it as a follow-up rather than forcing
     it in.
   - Handle the ?upgraded=1 success redirect with a brief confirmation
     message (the webhook may take a few seconds to land — don't assume
     planTier is already 'paid' the instant they land back here; either
     poll /api/receptionist/auth/me briefly or just show a generic "Thanks
     — this can take a moment to activate" message).
