CONTEXT — this is a scope change, read carefully before starting.

Up to now, the AI Receptionist signup form (/ai-receptionist/signup) only
captured a lead via POST /api/landing-test/submit — no real account was
created, and the confirmation was intentionally "you're on the list," not a
real login. That was correct for a smoke test. It no longer is: we're now
building the AI Receptionist as a real multi-tenant SaaS product with its
own customer accounts, completely separate from the internal CRM's admin
login. This task builds that foundation. It does NOT build Stripe billing,
hard trial-limit enforcement, or the CRM's admin oversight view of all
customers — those are separate, later tasks. Do not attempt them here.

Do not touch locked modules (routes/phone.ts, lib/discEngine.ts,
lib/leadScore.ts, lib/communicationIntelligence.ts, lib/workflowEngine.ts).
Do not touch the CRM's existing admin auth/session logic — this is a NEW,
separate auth system for AI Receptionist customers, not a change to how
SiteMint's own team logs into the CRM. Do not touch
/api/landing-test/view or page-view tracking.

BUDGET EXCEPTION (explicitly approved): schema migration + new auth routes

1. SCHEMA — extend intakeFirms for real accounts (lib/db/src/schema/intakeAgent.ts)

Add columns to the existing intakeFirms table:
- email (text, unique, not null) — the account's login email
- passwordHash (text, not null) — bcrypt hash, never store plaintext
- planTier (text, not null, default 'trial') — 'trial' | 'paid'
- trialConversationsLimit (integer, not null, default 20)
- createdAt (timestamp, default now) — if not already present

Any existing test firm rows (e.g. the seeded "AI for Lawyers (Test)" firm)
won't have email/passwordHash — either backfill a placeholder for that one
seeded row so the migration doesn't fail, or make the columns nullable and
enforce "not null" at the application layer for new signups only. Note
which approach you took in the Return report.

2. NEW AUTH — separate from CRM admin auth entirely

New file, e.g. artifacts/api-server/src/lib/receptionistAuth.ts:
- Session mechanism scoped to this product only — different cookie name
  than whatever the CRM admin session uses (check lib/auth or wherever the
  CRM admin session lives first, so there's no collision), e.g.
  "receptionist_session".
- requireReceptionistAuth middleware — validates the session, attaches the
  authenticated firmId to the request. A logged-in receptionist customer
  should only ever be able to query their OWN firmId's data — enforce this
  server-side in every new route below, not just in the UI.

New routes, e.g. artifacts/api-server/src/routes/receptionistAuth.ts:
- POST /api/receptionist/auth/signup — body: fullName, businessName, email,
  phone, industry, password. Creates the intakeFirms row (hash the
  password with bcrypt), sets the session cookie, returns the new account.
  Reject if email already exists (clear error, not a 500).
- POST /api/receptionist/auth/login — email + password, bcrypt compare,
  sets session cookie.
- POST /api/receptionist/auth/logout — clears session cookie.
- GET /api/receptionist/auth/me — returns the logged-in account's firm
  record (minus passwordHash) plus a count of their conversations so far
  (for the trial usage banner in step 4). 401 if not logged in.
