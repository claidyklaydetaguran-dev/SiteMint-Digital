TASK 2 — SERVER-SIDE PHONE NORMALIZATION ON LEAD UPDATE

Find the existing PATCH /api/crm/leads/:id endpoint.

When phone is included in the update payload:

* Run normalizePhone(phone)
* Store the normalized value
* If phone is empty/null, preserve current behavior
* Do not normalize unrelated fields
* Do not change schema

TASK 3 — LEAD DETAIL INLINE PHONE EDIT

In Lead Detail sidebar, add a small “Edit” action near the phone field.

Behavior:

* Clicking Edit turns phone into input field.
* Show helper text:
  “Use E.164 format when possible. US: +19498806515 · PH: +639186069624”
* Save button calls existing PATCH /api/crm/leads/:id with phone.
* Cancel restores original value.
* On save success:

  * refresh lead data
  * show toast
  * warning disappears if normalized correctly
* On save error:

  * show inline error

TASK 4 — SUGGESTED FIX BUTTON

If the warning detects a clear suggested format, show:
“Use suggested format”

Example:
09186069624 → +639186069624

Clicking it fills the input with the suggested value.
Do not auto-save without admin confirmation.
